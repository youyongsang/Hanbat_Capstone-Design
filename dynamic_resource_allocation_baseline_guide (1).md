# Dynamic Resource Allocation Model Baseline Guide

## 1. 문서 목적

이 문서는 현재까지 구현한 Reactive 기반 자원 할당 실험을 바탕으로, 다음 단계인 **동적 자원 할당 모델**을 어떤 방식으로 설계하면 좋은지에 대한 베이스라인 가이드를 정리한 문서이다.

본 가이드는 다음 목적을 가진다.

- Reactive 기반 자원 할당의 한계를 정리
- LSTM 기반 예측형 자원 할당 모델의 설계 방향 제시
- 입력 데이터, 출력 목표, 학습 방식, 평가 방식 제안
- 이후 강화학습(RL) 또는 고도화 모델로 확장 가능한 기본 구조 제공

---

## 2. 현재까지의 실험 결과 요약

현재까지의 실험에서 다음과 같은 사실을 확인하였다.

### 2.1 고정 자원 방식
- CPU를 고정한 상태에서 부하를 처리하도록 구성
- CPU를 일정 수준까지 올려도 latency 개선 효과는 제한적이었음
- p95 latency는 크게 줄지 않는 구간이 존재하였음

### 2.2 Reactive 방식
- 현재 상태를 보고 CPU 또는 컨테이너 수를 조절하는 방식
- 시스템 상태에 따라 자원을 동적으로 조정할 수 있었음
- 그러나 burst 트래픽 환경에서는 **이미 지연이 발생한 이후** 대응하는 구조이므로, 선제 대응에는 한계가 있었음

### 2.3 Hybrid Reactive 방식
- 먼저 CPU를 조정하고, 부족하면 컨테이너 수를 늘리는 방식으로 확장
- CPU scaling 단독보다 container scaling을 함께 사용했을 때 더 큰 개선이 나타났음
- 현재 실험에서는 컨테이너 수 5개, CPU 최대 3.0 수준에서 안정적인 성능을 확인함

---

## 3. 왜 Predictive / Dynamic Resource Allocation 모델이 필요한가

Reactive 방식은 다음과 같은 근본적 한계를 가진다.

1. **사후 대응 구조**
   - latency나 fail이 발생한 뒤에 자원을 늘림
   - 따라서 피크 초반에는 이미 성능 저하가 발생할 수 있음

2. **burst 트래픽에 취약**
   - 순간적인 트래픽 급증은 Reactive가 따라가기 전에 지연이 튈 수 있음

3. **과도한 진동 가능성**
   - threshold와 cooldown 설정에 따라 scale out / in이 불안정할 수 있음

따라서 다음 단계에서는 다음과 같은 모델이 필요하다.

> **과거 트래픽 패턴을 바탕으로 미래 부하를 예측하고, 미리 자원을 준비하는 예측형 동적 자원 할당 모델**

즉, 목표는 다음과 같다.

- 현재 상태를 보고 반응하는 Reactive를 넘어서
- **미래 부하를 예측하여 선제적으로 자원을 할당**하는 구조로 발전

---

## 4. 최종 목표 구조

동적 자원 할당 모델의 전체 구조는 다음과 같이 설계하는 것이 자연스럽다.

```text
과거 트래픽 시계열 데이터
→ 예측 모델 (LSTM)
→ 다음 시점 트래픽 또는 부하 예측
→ 자원 할당 정책
→ CPU / 컨테이너 수 결정
→ 실제 시스템 적용
```

즉, 모델을 크게 두 부분으로 나누는 것이 좋다.

### 4.1 예측 모델
- 과거 데이터를 기반으로 미래 부하를 예측

### 4.2 자원 할당 정책
- 예측값을 기반으로 CPU, 컨테이너 수 등 자원량을 결정

---

## 5. 1차 베이스라인 제안: LSTM 기반 트래픽 예측 모델

가장 추천하는 1차 베이스라인은 다음과 같다.

> **LSTM으로 다음 시점의 트래픽(RPS)을 예측하고, 예측된 RPS에 따라 자원량을 결정하는 구조**

이 방식이 좋은 이유는 다음과 같다.

1. 구현이 비교적 단순함
2. 해석이 쉬움
3. Reactive와 비교가 명확함
4. 이후 RL이나 멀티모달 자원 정책으로 확장 가능함

---

## 6. 입력 데이터 설계

### 6.1 가장 단순한 입력
1차 모델은 단순하게 시작하는 것이 좋다.

#### 입력 후보
- `target_rps`

즉, 과거 일정 구간의 `target_rps`만 사용하여 미래 RPS를 예측하는 방식이다.

예시:
- 입력: 최근 12초간의 `target_rps`
- 출력: 다음 1초의 `target_rps`

---

### 6.2 확장 가능한 입력
1차 모델 이후에는 아래와 같은 feature를 추가할 수 있다.

- `target_rps`
- `success_count`
- `fail_count`
- `avg_latency_ms`
- `p95_latency_ms`
- `sla_violation_count`
- 현재 CPU 사용률
- 현재 컨테이너 수

이렇게 하면 단순 트래픽 예측을 넘어서, 시스템 상태를 함께 고려하는 multivariate 시계열 모델로 확장 가능하다.

---

## 7. 출력(label) 설계

출력은 다음 두 가지 방식이 가능하다.

### 7.1 방식 A: 다음 시점 RPS 예측 (권장)
- 입력: 과거 N개의 트래픽 값
- 출력: 다음 1개의 `target_rps`

이 방식이 가장 기본적이고 해석도 쉽다.

#### 장점
- 순수 시계열 예측 문제로 분리 가능
- 자원 정책과 모델을 독립적으로 설계 가능
- 학습 및 평가가 단순함

---

### 7.2 방식 B: 자원량 직접 예측
- 출력: CPU 값 또는 컨테이너 수

예:
- 출력: `(cpu, replicas)`

이 방식은 최종적으로는 가능하지만, 현재 단계에서는 추천하지 않는다.  
자원 정책 기준이 좀 더 확립된 다음 진행하는 것을 추천한다.

---

## 8. 추천 모델 구조

### 8.1 입력 윈도우
예시:
- window size = 12

즉, 최근 12개 시점을 보고 다음 1개 시점을 예측한다.

### 8.2 출력 horizon
예시:
- horizon = 1

즉, 다음 1초의 부하를 예측

---

### 8.3 예시 형태

```text
입력 X: [t-11, t-10, ..., t]
출력 y: [t+1]
```

shape 예시:
- `X.shape = (samples, 12, 1)`
- `y.shape = (samples, 1)`

---

### 8.4 기본 LSTM 모델 예시

```python
model = Sequential([
    LSTM(64, input_shape=(12, 1)),
    Dense(32, activation="relu"),
    Dense(1)
])
```

loss 예시:
- `mse`

metrics 예시:
- `mae`

## 9. 자원 할당 정책 설계

LSTM의 출력은 예측 RPS이므로, 이를 CPU/컨테이너 수로 변환하는 **정책 함수(policy function)** 가 필요하다.

### 9.1 규칙 기반 정책 (권장)
초기 버전은 규칙 기반으로 시작하는 것이 가장 좋다.

예:

```python
def allocate_resource(pred_rps):
    if pred_rps < 150:
        return 0.5, 1
    elif pred_rps < 250:
        return 1.0, 2
    elif pred_rps < 400:
        return 2.0, 3
    else:
        return 3.0, 5
```

return값은 cpu와 컨테이너 수를 의미한다.

즉:
- LSTM = 예측 담당
- Policy = 자원 매핑 담당

---

### 9.2 왜 규칙 기반 정책이 좋은가
- 해석이 쉽다
- 디버깅이 쉽다
- Reactive와 비교가 쉽다
- 나중에 강화학습 정책으로 대체 가능하다

---

## 10. 실험 비교 구조

최종적으로는 아래 세 가지를 비교하는 구조가 가장 좋다.

### 10.1 고정 자원 방식
- 예: CPU 0.5, 컨테이너 1개 고정

### 10.2 Reactive 방식
- 현재 상태를 보고 CPU / 컨테이너 수 조절

### 10.3 Predictive 방식 (LSTM)
- 미래 RPS 예측
- 예측값 기준 선제 자원 할당

10.1과 10.2는 이미 구현된 사항이니 10.3만 만들면 된다.

## 12. 권장 구현 순서

### Step 1. 데이터셋 생성
- `target_rps` 기준 시계열 입력/출력 생성
- window size = 12
- horizon = 1

### Step 2. 단일 feature LSTM 학습
- 입력: `target_rps`
- 출력: 다음 시점 `target_rps`

### Step 3. 예측 결과 시각화
- actual vs predicted RPS 그래프

### Step 4. 예측값 → 자원 정책 연결
- 예측 RPS를 CPU/컨테이너 수로 변환

### Step 5. Predictive 방식 시뮬레이션
- 실제 시나리오에 적용

### Step 6. 고정 / Reactive / Predictive 비교
- latency, fail, SLA violation, 자원 효율 비교

## 14. 권장 1차 목표

현재 단계에서 가장 적절한 1차 목표는 다음과 같다.

> **과거 12초간 target_rps를 입력으로 받아, 다음 1초의 target_rps를 예측하는 LSTM 모델을 만들고, 예측된 RPS를 규칙 기반 자원 정책으로 변환하여 CPU와 컨테이너 수를 선제적으로 결정한다.**

이 정도면 다음을 모두 만족한다.

- 현실적 구현 가능
- 비교 실험 가능
- Reactive와 차별점 명확
- 이후 RL 확장 가능

## 15. 향후 확장 방향

### 15.1 Multivariate LSTM
단순 RPS뿐 아니라 latency, fail, CPU usage 등을 같이 입력으로 사용

### 15.2 직접 자원량 예측
출력을 RPS가 아니라 CPU/컨테이너 수로 직접 설정

### 15.3 강화학습(RL)
예측과 정책을 분리하지 않고, 상태를 보고 직접 최적 자원 행동을 선택

### 15.4 비용 포함 목적 함수
성능만이 아니라 자원 비용까지 고려

예:
- latency 최소화
- fail 최소화
- 자원 사용 최소화

---

## 16. 결론

현재까지의 실험 결과를 바탕으로 볼 때, 다음 단계의 가장 적절한 방향은 **LSTM 기반 예측형 동적 자원 할당 모델**이다.

Reactive 방식은 사후 대응이라는 구조적 한계를 가지므로, burst 트래픽 환경에서 완전한 성능 안정화를 보장하기 어렵다. 이에 따라 과거 트래픽 패턴을 바탕으로 미래 부하를 예측하고, 해당 예측 결과에 따라 CPU와 컨테이너 수를 미리 할당하는 방식이 더 적절하다.

따라서 본 프로젝트의 동적 자원 할당 모델은 다음과 같은 구조로 설계하는 것을 권장한다.

- LSTM으로 다음 시점 트래픽 예측
- 규칙 기반 정책으로 자원량 결정
- Predictive 자원 할당 시뮬레이션 수행
- 고정 자원 및 Reactive 방식과 비교 평가
